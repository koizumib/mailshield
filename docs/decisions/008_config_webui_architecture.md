# 008: 設定（ポリシー・ワーカー・ルーティング・変数）の WebUI 化アーキテクチャ

## 決定

ポリシー・ワーカー・ルーティングの設定を、サーバ上のテキストファイル＋ディレクトリ命名順から、
**WebUI から編集できる「名前付き部品の合成モデル」** に移行する。設定の真実の源は
MariaDB の**バージョン付き設定ドキュメント**とし、smtp-gateway は起動時にインメモリへ読み込んで
以後メール処理経路では DB を引かない。ファイルベース運用も同一モデルで維持する。

本 ADR は ADR 002（after-queue content filter）・ADR 005（キューを使わず「DB が真実＋
ポーリング＋CAS」・必須は MariaDB のみ）の思想の上に構築する。

### 1. 設定モデル：4 つの名前付きエンティティ ＋ 合成

| エンティティ | 内容 | 参照関係 |
|---|---|---|
| **ワーカーインスタンス** | ワーカー型（コード側の実装。filesep / av-worker / Lua 等）＋その設定＋名前 | ルーティングから参照される |
| **ポリシー** | 検査結果 → アクションのルール群（現行 policy.yaml 相当） | ルーティングから参照される |
| **ルーティング** | priority・match 条件・inspect/transform のインスタンス束ね（順序・有効無効・timeout 上書き）・policy 参照 | 上 2 つを合成する |
| **変数（env）セット** | key→value の共有値 | 上 3 種の設定内から `${VAR}` で参照される |

- 現行の `routes.d/<route>/{route.yaml, policy.yaml}` が「ルーティング＋ポリシー」に、
  `config/workers/<name>.yaml` が「ワーカーインスタンス」に対応する。
  **新概念はワーカーインスタンスを route から分離し、名前付き・再利用可能にした点のみ。**
- ワーカー**型**（Go/Lua の実装）はコード側で登録され不変。ワーカー**インスタンス**は
  「型＋設定＋名前」のデータであり WebUI で任意個作れる。同じ型から用途別インスタンスを
  複数定義できる（例: 内部向け添付分離 / 外部向け添付分離）。
- **ポリシーも名前付き部品**とし、複数ルーティングで共有・単体でエクスポートできる。

### 2. 識別：id / alias / display_name の 3 層

ワーカーインスタンス（およびポリシー等の参照される部品）は 3 フィールドを持つ。

- `id`（UUID・不変・内部参照の実体）
- `alias`（短い安定ハンドル・rename-safe。**条件 DSL と検査結果のキーに使う**。例 `av_internal`）
- `display_name`（日本語可・画面表示専用・いつでも変更可）

**検査結果はワーカー型名ではなく alias でキーする。** ポリシー条件も alias 参照（例
`av_internal.detected == true`）。UUID を DSL に書くと読めず、display_name をキーにすると
改名で条件が壊れるため、3 層に分ける。同一ルーティングに同型インスタンスを複数束ねる場合も
alias が別なので衝突しない（保険として「バインディング側で結果キー上書き可」とする）。

### 3. ルーティングの評価

- **1 メール = 1 ルーティング（first-match）。** priority 昇順で最初に match したルーティングだけを通す。
  「全マッチを通す」は transform 合成が破綻するため採らない。
- **catch-all を必ず自動保証する。** システムが「match=true・最低 priority」のルーティングを
  1 つ保証（削除・並べ替え不可の固定枠）し、マッチ漏れによるメール消失を構造的に防ぐ。
  ただしその catch-all が使う policy／ワーカー束ねは自由に設定できる（ポリシーの
  `default_deliver` 自動挿入と同じ思想：枠は固定・挙動は選択可）。
- **match 条件と policy 条件は式エンジン（文法・演算子）を共通化し、使える事実集合を
  フェーズで分ける。** match はワーカー実行前なので封筒／ヘッダのみ、policy はワーカー実行後なので
  スコアも使える。※現行の route match は `match.to`/`match.from` のフィールド指定のみで、
  本 ADR で式ベースへ一段進化する。
- bounce ルーティングは「MAIL FROM 空」を match で表現する（特別扱いしない）。
- timeout は「インスタンス既定値 ＋ ルーティング側で上書き可」。

### 4. 保持方法：DB がバージョン付きドキュメントの真実の源

- **真実の源 = MariaDB。** 設定は正規化列ではなく**バージョン付き設定ドキュメント（JSON blob）**として
  保持する。`config_versions {version_id, content, checksum, author, created_at}` ＋
  `config_active {version_id}` の 1 行ポインタ。既存 `policy_versions` の一般化。
  - RDB 正規化しない理由：個々の設定構造はアップデートで頻繁に変わるため、列で持つと
    スキーマ追随が破綻する。構造化ドキュメントをそのまま格納し、**検証はアプリ層で全体一括**に行う。
- **実行時 = 各 smtp-gateway が起動時にアクティブ版をメモリへ読み込みパイプラインを構築。
  メール処理経路では DB を一切引かない。** 「ファイルをメモリに読む」を「DB から読む」に替えるだけ。
- **複数レプリカへの伝播：**
  - **ベースライン = ポーリング。** 各 gateway が「アクティブ版ポインタ（version_id＋checksum の 1 行）」を
    数秒間隔で確認し、差分があれば全設定を読み直してメモリを原子的に差し替える。
    1 レプリカあたり数秒に 1 回の極小クエリ（メール毎ではない）。追加インフラ不要（MariaDB のみ）。
  - **任意の高速化 = Redis pub/sub。** 更新時に通知を publish して即リロード。あくまでオプションの
    加速装置とし、Redis 無しでもポーリングで回る（「必須は MariaDB のみ」を守る）。
  - アクティブ版ポインタの単一行更新が合意点（consensus）となり、切替は原子的・
    ロールバックはポインタを戻すだけ。この規模で etcd 級の合意機構は不要。

### 5. canonical model と ConfigSource：file／DB の共通インタフェース

- **正規モデル（canonical config model）を 1 つ持ち、`ConfigSource` インタフェースの背後に
  file 実装と DB 実装の 2 つを置く。** 両者は同一のインメモリ・スナップショットを生成する。
- **manifest/bundle 形式を唯一の交換形式＝ファイル形式とする。** これにより file ⟷ DB 変換は
  manifest を介して自明になり、「DB 運用とファイル運用で構造が変わらない共通インタフェース」を実現する。
- **設定ソースは deploy 時に選択（`file` / `db`）。**
  - `file` モード：ファイルが真実。WebUI 編集はドラフト（DB 保持）→「テスト適用」で稼働中インスタンスへ
    ephemeral 反映（ポーリング／pubsub 経由・再起動で消える）→ 良ければ**エクスポートしたバンドルを
    基盤の config ディレクトリに配置し恒久化**。エクスポート出力は FileConfigSource が読む形式と同一。
  - `db` モード：DB が真実。activate でアクティブ版ポインタ切替・ロールバック可・完全動的。

### 6. インポート／エクスポート：k8s マニフェスト風バンドル

- すべてのエンティティを `{kind, name, spec}` の 1 ドキュメントで表し、**参照は名前で**行う。
- **バンドル = ドキュメントの配列。** 含めるドキュメントを選ぶだけで粒度は自由
  （1 インスタンスだけ／ルーティング＋依存一式／全部）。
- エクスポートに「**依存も含める**」トグルを設け、「このルーティングだけ」でも「参照先ごと」でも出せる。
- インポートは `(kind, name)` で**冪等 upsert**。参照先が欠けていればエラー（またはバンドル同梱）。
- デフォルト設定（現行 bounce/inbound/outbound）もバンドルとして同梱し、初期テンプレのインポートで表現する。

### 7. 変数（環境変数）

- **変数セットを第一級エンティティ**とし、ルーティング match・ポリシー条件・ワーカー設定・配送先などから
  `${VAR}` で参照する。**展開は設定ロード時（スナップショット構築時）に一度だけ**行い、メール毎の
  実行時コストはゼロ。
- **2 種類を厳格に分ける：**
  - **設定変数**（ドメイン・閾値など非機密・環境依存）：WebUI で管理し DB に格納。エクスポート対象だが
    **値は環境依存のためバンドルには載せず、`requires_variables` で必要な変数名だけ宣言**し
    インポート先で入力させる。
  - **シークレット**（パスワード等）：DB にも設定にも入れず **OS 環境変数のまま**。設定からは名前参照のみ。
    エクスポートに絶対に混ぜない。
- **反映タイミング：** WebUI 管理の設定変数を変更すると「レンダリング後スナップショットが変わる」＝
  新しい config 版が生まれるため、他の設定変更と同一経路で反映される（ポーリング間隔＝ほぼ即時、
  pubsub 併用なら即時）。**OS env のシークレットはプロセス起動時にしか読まないため restart が必要。**

### 8. 検証と原子性

版を切り替える前に**バンドル全体を一括検証**する：ルーティングが実在するインスタンス／ポリシーを
参照しているか、policy 条件が参照する alias が当該ルーティングの inspect 集合に存在するか、
参照切れ・catch-all 欠如・未定義変数が無いか。すべて通って初めてアクティブ版ポインタを更新する。
これが「DB を正規化列でなくドキュメント＋アプリ層検証」にする最大の理由。

## 理由

- 現行のファイル＋ディレクトリ命名順は WebUI から安全に触れない（基盤ファイル・ディレクトリ構造を
  WebUI から直接操作させるのは Docker 権限・セキュリティ上不適切）。
- 個々の設定構造はアップデートで変わるため RDB 正規化はスキーマ追随リスクが高い。ドキュメント＋
  バージョニングは既存 `policy_versions` で実績があり、それを一般化するだけで済む。
- メール処理経路で DB を引かない（インメモリ・スナップショット）ことで、スループットと
  「MariaDB のみ必須」を両立できる。伝播はポーリングで足り、ADR 005 と一貫する。
- canonical model ＋ manifest 共通形式により、file 運用と DB 運用・インポート／エクスポート・
  デフォルトテンプレを 1 つの形式で統一でき、実装とユーザー体験の分岐を最小化できる。

## 段階導入

1. **ワーカーインスタンス ＋ 変数**（土台：canonical model の型・CRUD・管理 UI）
2. **ルーティング**（priority・match 式・インスタンス束ね・policy 参照・catch-all 保証）
3. **DB 版 ConfigSource ＋ ポーリング ＋ インポート／エクスポート**（gateway をインメモリ・
   スナップショット駆動へ。file/db 切替・manifest 入出力・Redis pub/sub は任意加速）

ポリシー編集 UI は既存実装（`policy_versions`・ロールバック）を本モデルへ合流させる。

## UI マニュアルの方針（必須）

設定が Zabbix 的に独自概念を増やすため、各画面のヘルプ（`web/src/help/content.ts`）は
「概念（位置づけ）／用語集／初回セットアップ手順／よくある例／落とし穴（catch-all 必須・
変数の展開タイミング・alias 参照・first-match 等）／ガイドツアー／空状態の導線」をセットで用意する。
各画面の完成条件（Definition of Done）にこのヘルプ整備を含める。
